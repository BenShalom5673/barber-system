## Availability Rules

### Time Alignment
- **All customer-facing appointment times must be on 5-minute increments.**
  - Valid: 09:00, 09:05, 11:25, 12:50, 13:15
  - Invalid: 09:03, 11:23, 12:52
- All service durations must also be multiples of 5 minutes. This keeps the system clean.

### Slot Grid Interval
- The `slotIntervalMinutes` setting controls the fixed grid density (how often candidates are generated within working hours).
- **This value must be 5** â€” matching the product rule above.
- âš ď¸Ź Current risk: the database default and the settings service default are both **15**. This must be corrected. Until corrected, the fixed grid may run at 15-minute density when a barbershop settings row exists (though gap-based candidates still fill the gaps at finer resolution).

### Gap-Based Scheduling
- After any existing appointment ends at time T, the next available slot may start at exactly T â€” not rounded to the next grid time.
- From T, additional slots are generated by stepping forward using the **service duration** (not including buffer).
  - Example: appointment ends at 12:25, service = 25 min â†’ gap candidates: 12:25, 12:50, 13:15, 13:40 â€¦
- Only candidates that are 5-minute aligned (i.e., `millisecondsSinceEpoch % 300000 === 0`) are included.

### Buffer Time
- After every appointment, a **buffer** is added (default: 5 minutes, configurable per barbershop via `barbershopSettings.appointmentBufferMinutes`).
- The buffer is included in the stored slot range (`slotRange`) in the database.
- The buffer is **never shown to customers** â€” it is invisible.
- The buffer prevents two consecutive bookings from touching with zero gap.

### Working Hours
- Each barber has a weekly schedule stored in `staff_working_hours`.
- Day numbering: 0=Sunday, 1=Monday, 2=Tuesday, 3=Wednesday, 4=Thursday, 5=Friday, 6=Saturday.
- Availability is only generated within the barber's working window for that day.
- A `day_off` override in `staff_schedule_overrides` cancels all availability for a specific date.
- A `custom_hours` override replaces the weekly schedule for a specific date.
- A `blocked_slot` override blocks a specific time range on a specific date.

### Overlap Check
- A proposed slot is rejected if `[slot_start, slot_start + duration + buffer)` overlaps any existing blocked range.
- Half-open intervals are used: touching boundaries do NOT count as overlapping.
  - Example: existing slot ends at 12:25, new slot starts at 12:25 â†’ they do NOT overlap. Correct.

## Booking Rules

### Required Fields
- Every booking requires: staff member, service, time slot, customer name, customer phone.
- Email and birthday are optional (used for marketing consent).

### Customer Matching
- Customers are matched by phone number (stored in E.164 format). Same phone = same customer record. No duplicates.

### Customer Status
- `active` or `vacation`: can always book online.
- `restricted`: can only be booked manually by the owner. Blocked from the online flow.
- `irrelevant`: cannot book at all, in any context.

### Slot Re-Validation
- When the customer clicks "Confirm", the backend re-checks that the slot is still available before inserting.
- If the slot was taken in the meantime, the response is a `409 Conflict` error.
- The database GiST constraint is the final race-condition safety net even after re-validation.

### Immutable Snapshots
- When a booking is created, the price, duration, and service name are copied into the appointment record.
- Future changes to the service do NOT affect existing bookings. This is intentional.

### Appointment Status Flow
- New bookings start as `confirmed` (if no deposit required) or `pending_deposit`.
- Valid statuses: `pending_deposit`, `confirmed`, `completed`, `cancelled`, `no_show`.

## Staff Rules

### Service Assignments
- If a staff member has **no service assignments** in `staff_services`, they are eligible for ALL services.
- If a staff member has **any assignments**, they are only eligible for services explicitly listed.
- Adding even one assignment to a staff member restricts them from all other services.

### Dev Seed Rule
- For testing purposes only, the seed script links ALL active staff to ALL active services.
- In production, the barbershop owner controls which services each barber offers.

### Soft Delete
- Staff members are never deleted â€” they are deactivated (`isActive = false`). Historical appointments still reference them.

## Service Duration and Buffer Rules

- Service duration = actual time with the customer. Does NOT include buffer.
- Buffer = prep time after each appointment. Default is 5 minutes.
- `slotRange` stored in DB = `[start, start + duration + buffer)`.
- What customers see = `start` to `start + duration` only (buffer is hidden).
- Duration displayed to customer = `service.durationMinutes` (never includes buffer).

## Database Rules

### Double-Booking Prevention
- A PostgreSQL GiST exclusion constraint prevents two non-cancelled/non-no_show appointments for the same barber from having overlapping slot ranges.
- This is the last line of defense against race conditions.

### Slot Range Format
- Stored as `tstzrange` (PostgreSQL range type with timezone).
- Example: `["2026-04-27T09:00:00.000Z","2026-04-27T09:30:00.000Z")` â€” always UTC, half-open.

### Prices
- All prices are stored in **agorot** (integer). 1 shekel = 100 agorot.
- Always divide by 100 when displaying to customers. Never use floating-point numbers for money.

### Migrations
- Schema changes must go through Drizzle migrations (`npm run db:migrate`).
- Never edit the database directly for schema changes.
- Migration files in `src/server/db/migrations/` are permanent â€” never delete or edit them.

### Foreign Key Delete Behavior
- Appointments reference customers, staff, and services with `onDelete: 'restrict'` â€” deleting a customer/staff/service is blocked if they have appointments.
- Barbershop deletion cascades to everything (nuclear option).

## UX Rules

These are product principles that guide all UI and flow decisions. Full details in PROJECT_MASTER.md under "Product Principles".

- **The system leads, not asks.** Never present open-ended choices when the system can make a smart default decision and present it.
- **Fewer choices â†’ higher conversion.** Reduce options at every step. Show what the user needs, hide what they don't.
- **Every action must be clear, fast, and ideally one click.**
- **The user sees only what they need.** Complexity belongs in the backend, not the UI.
- These principles override UX decisions when conflicts arise.

## Code Rules

### TypeScript Strictness
- `exactOptionalPropertyTypes: true` â€” cannot pass `string | undefined` to a field typed `string | null`. Use `?? null` to convert.
- `noUncheckedIndexedAccess: true` â€” array index access returns `T | undefined`, not just `T`. Always handle the undefined case.

### No Rounding of Times
- Never round slot times to 15-minute or 30-minute grids. The system uses 5-minute resolution only.
- Never hardcode interval logic using 15 or 30 as a constant.

### Timezone
- All timezone-sensitive logic must use `Asia/Jerusalem` (via the `DEFAULT_TIMEZONE` constant in service files).
- Never assume UTC = local time.
- The `barbershops.timezone` column exists but is NOT yet used by the availability engine. Do not use it without updating the engine too.

### API Protection Pattern
- Public booking flow routes (GET services, GET staff, GET availability, POST bookings): no token required â€” customers call these.
- Admin/owner routes (create/edit services, staff, settings): require `guardInternalToken()` header check.
- Never add `guardInternalToken` to routes that customers must access.
